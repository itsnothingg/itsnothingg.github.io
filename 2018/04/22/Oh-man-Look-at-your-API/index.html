오 이런, 당신이 만든 API 좀 보세요!

itsnothingg Apr 22, 2018
  • Share to Weibo
  • Share to Twitter
  • Share to Facebook
  • Share to Google+

해당 포스트는 Alexey Kuznetsov의 Oh Man, Look at Your API! 포스트를 번역한 것입니다.

Oh Man, Look at Your API!

세상을 좀 더 나은 곳으로 만들기 위한 저의 작은 노력이 여기 있습니다. API를 구축하고 있거나 기존의 API를 개선하고 싶은 마음이 있는 분들에게 유용하리라 생각합니다. 여기 나오는 조언들은 제가 작업했던 수 많은 API들에서 겪은 괴로운 경험에 기초하고 있습니다. 치를 떨만큼 싫은 API나, 능력을 개발하는 데 도움이 되기도 하는 API들이 있었지만, 대부분은 좌절감과 허망한 웃음정도 밖에 남기지 않았습니다. 보다시피 다뤄야 할 것들이 많으니 시작해봅시다!

잘 만들어진 API는 심심합니다. 예외는 없습니다.

어떠한 새로운 것을 개발해내려 하지 마세요. 시스템의 성질은 어디서나, 어떤 단계든지 동일해야합니다. API 끝점 구성 및 데이터 입력/출력 형식에 모두 적용되는 원리입니다. 우리는 심심한 표준 API오류 메시지와 요청/응답 헤더가 필요합니다. 모든 데이터 타입의 필드는 항상 입력/출력 형식이 동일해야 합니다. 공백인 필드는 모두 동일하게 표시되어야 합니다. API 프로세스를 프로그래밍하는 것은 거짓말하는 것과 매우 유사합니다. 많은 것들이 쌓이면 쌓일수록, 나중에 여러분들이나 사용자분들이 감당해야 할 것들도 많아집니다. 다른 말로 하면, API가 더 심심하게 설계 될 수록 사용자분들의 만족도는 올라가게 되어있다는 것입니다.

아, 이 조언은 전체 정보 시스템을 설계 할 때도 적용되니, 마음에 새겨두는 것이 좋습니다.

문서화

우리는 늘 API를 문서화 하라는 조언을 듣지만, 대부분의 사람들은 이 일을 하기 싫어합니다. 문서화 할 시간이 충분해 보이는 사람이 아무도 없다는 것을 이해하지만, 문서는 API의 얼굴입니다. 코딩을 잠시 멈추고, 문서를 작성하는 데 시간을 투자하는 것이 더 가치 있을 수 있습니다. 문서화를 위한 몇가지 기본 팁이 있습니다.

  • 동작하는 HTTP 요청 및 응답 예제나, 실제로 사용되고 있는 코드를 제공해야합니다.
  • 문서는 API 자체 만큼이나 일관성있게 서식화 되어야 합니다.
  • API의 기본 원칙을 명확하고 자세히 설명해야 합니다.
  • Swagger, Apiary 또는 Postman과 같은 API프레임워크를 사용해야 합니다.

좋은 문서를 제공하지 않는 API를 사용하는 것은, 마치 복잡한 퍼즐을 조각 조각 맞추는 것과 유사한 경험을 겪게합니다.

오류 처리

문서화가 잘 되어 있지 않더라도 오류 메시지가 잘 되어있다면 그것에 의존하여 꽤 많은 것들을 처리 할 수 있습니다. 사용자 경험에 있어서 명확하고 정확한 오류 메시지를 전달 받는 것이 굉장히 중요하다는 것은 두말하면 입아픈 사실입니다. 마찬가지로 API 개발자들에도 좋은 오류 메시지는 작업에 대한 자신감을 높여 주고 오류에 신속하게 대응할 수 있도록 해 줍니다.

잘못된 오류:

HTTP/1.1 422 Unprocessable Entity
{
 “error”: Invalid Attribute
}

좋은 오류:

HTTP/1.1 422 Unprocessable Entity
Content-Type: application/vnd.api+json
{
 "errors": [
 {
 "status": "422",
 "source": { "pointer": "/data/attributes/first-name" },
 "title": "Invalid Attribute",
 "details": "First name must contain min three chars."
 }
 ]
}

Authorization

오직 표준화 된 Authorization 만이 사용 가치가 있습니다. 부디 이를 망치지 말아주세요. Authorization 프로세스가 너무 독특하거나 너무 취약하면 심각한 문제가 됩니다. OAuth를 예로 들어봅시다. OAuth는 액세스 토큰이 업데이트 될 때마다 리프레시 토큰을 발급합니다. 또한 액세스 토큰에는 만료 시간도 있어야 합니다. 제 생각에 이건 너무 과다한 것 같습니다. 첫째로 사용자가 요구하는 보안 수준을 파악해보고, 만료되지 않는 토큰을 사용해도 되는 지 판단해 볼 필요가 있습니다. 둘째로 프로세스를 단순화 시켜야 합니다. 많은 훌륭한 API들이 정적인 리프레시 토큰을 사용합니다. 보안성과 복잡성이 동의어는 아닙니다.

데이터 형식

새로운 데이터 형식을 구성할 생각조차 하지 말아주세요! 고유한 데이터 형식을 사용하는 API를 몇 번 본적있었는데, 늘 엉망이었습니다. 고유한 데이터 형식을 사용하면 작업 속도가 크게 저하되고 처리할 일이 많아집니다. 물론, 이런 이상한 생각들이 쏟아질 때 여러분들의 창조성을 억누르기는 힘들다는 것을 알고 있고, 힘들게 개발한 작업물을 그냥 던져 버리는 것이 매우 어렵다는 것도 잘 알고 있지만, 꼭 저항해야합니다. 우리에게는 이미 JSON, XML과 다른 모든 것들이 있습니다. 그렇죠? 그것들이면 충분합니다.

JSON API와 같은 좋은 사양을 이용해야합니다. JSON API는 Stripe, Braintree등의 많은 회사에서 이미 성공적으로 사용하고 있습니다. JSON은 보기에 좋으며, 구조화된 데이터 형식을 사용하기 쉽고, 이미 대중화 되어 표준 규격으로 자리 잡았습니다.

아 그리고, 다중 데이터 형식을 지원할 필요가 없습니다. 모든 프로그래밍 언어 및 기술은 이미 모든 표준 유형에 대해 데이터 형식 지정 및 구문 분석을 지원하기 때문에 사용자들은 일반적으로 어떤 표준 데이터 형식을 사용하는지 크게 관심이 없습니다. 다중 데이터 형식을 지원 하면 API 및 문서 유지 관리 속도만 느려집니다.

HTTP 상태 코드

HTTP 상태는 200인데, 메시지는 내부 API 오류를 전달하는 것은 개발자들에게 혼란을 야기합니다. 이런 일은 개발을 지연시킵니다. HTTP 상태 코드는 오류 메시지 역할을 대신할 수 있습니다. 다음과 같은 일반적인 코드 목록이 있습니다.

200 (OK) — success, handle responded data.
500 (Internal Server Error) — something went wrong, ask for API system technical support.
400 (Bad Request) — wrong API request, check documentation.
401 (Unauthorized) — wrong client credentials, recheck them or if API key token has expired, then refresh.
403 (Forbidden) — unpermitted action, current user does not have enough access rights.
404 (Not Found) — record not found, maybe it doesn’t exist anymore or wrong record ID was used.
405 (Method Not Allowed) — this API method is not allowed anymore.
406 (Not Acceptable) — not acceptable action, probably from attempt to save invalid data.

위키피디아에서 남은 코드 목록들을 확인해 보면 틀림없이 언제나 적절한 코드를 찾을 수 있을 것입니다.

Content-type

매우 간단한 조언입니다. 변경해야 할 특별한 이유가 없다면 Content-type은 항상 같아야 합니다. 사용자 아키텍처의 일관성을 유지시키고 개발 프로세스를 가속화 및 단순화할 수 있도록 도움이 됩니다. 끝점에 대해 여러개의 Content-type 을 지원하지 마세요. 끝점을 사용하여 작업하는 개발자는 이 때문에 혼란스러울 수 있습니다. 예를 들어 끝점”A”는 정상적으로 작동하지만 끝점”B”는 동일한 방식으로 사용하더라도 잘못된 요청 오류를 반환 하는 경우가 생길 수 있습니다. Content-type을 변경하는 것은 명확한 해결책으로 보이지 않습니다.

객체 구조

사용자 입장에서 볼 때, 객체가 실제로 어떤 개체들을 포함 하는지 알 수 있는 것은 매우 도움이 됩니다. API에 전체 개체 필드 맵을 반환하는 특수한 끝점이 있다면 완벽합니다. 그런 끝점이 없다면 최소한 객체를 호출할 때, 개체가 비어 있는 경우에도 모든 필드를 반환하도록 보장해야합니다. 여기서 역동성과 간결함은 요구되지 않습니다.

{
 "id": "9c7fd1e5-97f0-4b6a-979c-1d7adf5bda16",
 "email": "example@gmail.com",
 "firstname": "John",
 "lastname": "Smith",
 "timezone": null,
 "phone_code": "RU",
 "phone": null,
 "created_at": "2018-03-27T05:40:31.929Z",
 "updated_at": "2018-04-08T09:24:39.683Z",
 "citizenship": null
}

Pagination

제가 한 유명한 CRM시스템에서 API Connector를 만들고 있을 때, 그 시스템에 Pagination 리코드가 없는 것을 발견하여 관련해서 기술 지원에 문의를 한 적이 있습니다. ”불행하게도 포함되지 않는 기능입니다. 하지만 좋은 아이디어네요” 와 같은 믿을 수 없는 답변을 들어야 했습니다. 저는 여러분 모두가 왜 우리가 모든 데이터를 동시에 로딩할 수 없는지 명확하게 이해하고 있다는 것을 알고 있기 때문에, 제 관점에서 무엇이 완벽한 Pagination인지 설명해 드리겠습니다.

  • 페이지 크기를 제어할 수 있는 것은 유용합니다. 사용자가 스스로 요청의 양과 품질 사이에서 선택할 수 있도록 기회를 제공해줍니다.
  • 얼마나 많은 레코드를 갖고 있는지 이해하도록 메타 정보를 가지고 있습니다. 모든 레코드를 로딩하는 경우 진행 상황을 파악하는 데 큰 도움이 됩니다. 또한 레코드 수를 계산하는데 모든 페이지를 로딩할 필요가 없으므로, 시간과 돈을 아낄 수 있습니다.
  • 명확하고 일관성있는 pagination 변수명이 선언됩니다. (예:page_number-현재 페이지 번호, page_size — 페이지 단위, pages_total_count — 총 페이지 수 등).

이미지 및 파일

당연한 것이지만, 응답 데이터에 파일이나 영상 본문을 포함해서는 안 됩니다. 임시 업로드 링크를 반환하는 것이 훨씬 낫습니다.

시간과 날짜

사용되는 표준 시간대를 명시적으로 지정하는 것이 정말 중요하다는 것을 짚고 싶습니다. 문서에 명시할 수도 있고, 데이터 형식에 명시할 수도 있습니다. 시간대를 어떻게 사용해야하는지는 여기서 다룰 만한 내용이 아니지만, 간단하게 말하자면: 유닉스 시간을 반환하면 됩니다. 유닉스 시간은 UTC 시간대를 기준으로 하기 때문에, 모든 사용자들은 이 유닉스 시간을 받아 자신에게 알맞은 형식과 시간대로 변환 할 수 있습니다.

{
 …
 "created_at" => article.created_at.utc.to_i 
}.to_json
#=> { …, created_at: 1523556863 }

결제 구상

미결제 사용자가 결제를 필요하는 끝점을 사용하려고 할 때 HTTP상태 코드를 사용하여 사용자에게 결제가 필요하다는 것을 알려야 합니다. 때로 사용자는 무엇이 잘못된 줄 모르고 애꿎은 곳에 시간을 낭비하는 경우가 생길 수 있습니다. 여기에는 402 “Payment Required” HTTP상태 코드를 사용하시기를 권합니다. 사용자에게 명확하게 전달만 된다면 어떤 HTTP 상태 코드를 사용하는지는 크게 상관 없습니다. API문서에도 이러한 끝점 사용 조건에 사항이 명시 되어 있어야 합니다.

API 요청 제한

API에 요청 제한을 두기를 원하신다면, 한계에 도달했을 때 명확한 오류 메시지가 표시되어야 합니다. 이상적으로, 이러한 제한 관련 정보를 포함하는 특수 끝점이 있다면 좋습니다. 사용자가 API요청 수를 계산할 필요가 없어야 합니다. API 쪽에서 모든 정보를 이미 갖고 있기 때문입니다. HTTP응답 해더에 사용 가능한 API호출에 대한 정보를 제공하는 것이 좋습니다. 예를 들어:

X-RateLimit-Limit - Request limit per hour
X-RateLimit-Remaining - The number of requests left for the time window

Example of API Limits in ZohoCRM

논리적인 측면

API를 개발하는 동안 시행할 수 있는 좋은 아이디어 중 하나는, 자신을 사용자와 동일한 자리에 올려 놓는 것입니다. 이렇게 하면 초기 단계에서 많은 유용한 것들을 발견할 수 있습니다. 예를 들어 저는 일부 API에서 동일한 레코드를 두 번 삭제해도 똑같은 응답이 나타난다는 것을 발견했습니다. 제 생각에 이 응답은 사용자가 레코드를 다시 제거했거나 다른 레코드를 제거했다는 잘못된 인식을 갖게 하기 때문에 좋지 않습니다. 기록이 이미 삭제된 경우 404 HTTP상태 코드를 반환하여 고객이 실수를 알아차릴 수 있도록 해 주어야 합니다.

API에 대한 고찰

이상한 API 끝점이 있다는 것은 나쁜 프로젝트 설계의 징후일 수 있습니다. 그러니, facades나 adapter를 만드는 데 뛰어들지 말고, 잠시 코드에서 눈을 때고 큰 그림을 둘러보세요. 리팩터링의 좋은 기회가 되어 API 인터페이스를 단순화하고, 프로젝트 구조를 개선 시킬 수 있는 좋은 기회가 될 가능성이 있습니다.

마치며

뛰어난 품질과 사용하기 쉬운 API를 만들어 내는 모든 개발자들에게 큰 감사를 전하고 싶습니다! 여러분들은 주로 자기만의 인식과 감각에 기인하여 행동한다는 것을 알고 있습니다. 얼핏 보면, 이러한 투자는 단기적으로 명백한 이익을 내지 못하는 것처럼 보일 수 있지만, 점점 늘어나는 고객 충성도를 통해 장기적인 효과를 거둘 것이라고 믿고 있습니다. 이 고된 일을 부지런한 하는 개발자들에게 행운이 있기를!
